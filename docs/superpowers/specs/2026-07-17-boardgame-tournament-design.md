# เอกสารดีไซน์: เว็บจัดการแข่งขันหมากกระดาน (Boardgame Tournament)

วันที่: 17 กรกฎาคม 2026
สถานะ: ผ่านการอนุมัติดีไซน์จากผู้ใช้แล้ว (รอรีวิวเอกสาร)

## 1. เป้าหมาย

เว็บสำหรับจัดการแข่งขันหมากกระดาน 7 หมวด ได้แก่ หมากรุกไทย, หมากรุกสากล, หมากล้อม, หมากฮอส, ครอสเวิร์ด, เอแมท และบอร์ดเกม (หมวดรวมสำหรับบอร์ดเกมอื่น ๆ) โดยผู้จัด (กรรมการ) ใช้บันทึกผลการแข่ง ชนะ/เสมอ/แพ้ ระบบรวมคะแนนสะสมและจัดอันดับให้อัตโนมัติ ผู้ชมเปิดดูผลจากเครื่องใดก็ได้แบบเรียลไทม์

ผู้ใช้หลัก: ผู้จัดการแข่งขันระดับโรงเรียน/ชมรม (ไม่ต้องมีระบบสมัครสมาชิก)

## 2. ขอบเขตความต้องการ (Requirements)

- แยกหน้าแต่ละหมวดเกม ทั้ง 7 หมวด แต่ละหมวดสร้างทัวร์นาเมนต์ได้หลายรายการ และเก็บประวัติรายการที่จบแล้ว
- ผู้เข้าแข่งต่อทัวร์นาเมนต์: 1–50 คน ระบุด้วยชื่อ (ไม่มีทะเบียนผู้เล่นกลาง)
- ระบบจับคู่ 4 แบบ: บันทึกอิสระ, พบกันหมด (round robin), สวิส (Swiss), น็อกเอาต์ (knockout)
- ผลการแข่ง: ชนะ / เสมอ / แพ้ ต่อแมตช์ และกรอกคะแนนดิบในเกมได้ (ไม่บังคับ) เช่น เอแมท 320–250
- แต้มสะสม ชนะ/เสมอ/แพ้ ตั้งค่าได้ต่อทัวร์นาเมนต์ (ค่าเริ่มต้น 1 / 0.5 / 0)
- ตารางคะแนนสะสมและอันดับ อัปเดตเรียลไทม์ทุกเครื่อง
- การแก้ไขข้อมูลต้องใส่ PIN ของทัวร์นาเมนต์ (โหมดกรรมการ); คนทั่วไปดูได้อย่างเดียว
- UI สองภาษา ไทย/อังกฤษ สลับได้ จำค่าที่เลือกไว้ในเครื่อง

### แบ่งเฟส

- **เฟส 1 (ใช้งานจริงได้):** โครงเว็บทั้งหมด, ทุกหน้า, ระบบ PIN, i18n, ระบบจับคู่แบบอิสระ + พบกันหมด, ตารางคะแนน/อันดับ
- **เฟส 2:** ระบบสวิส + น็อกเอาต์ (พร้อมแผนภาพสายแข่ง)

โครงสร้างข้อมูลรองรับทั้ง 4 ระบบตั้งแต่เฟส 1 (ฟิลด์ `format` มีครบ) เพียงแต่ UI ของสวิส/น็อกเอาต์มาทีหลัง

## 3. สถาปัตยกรรม

- **Frontend:** React + Vite (SPA), ภาษา TypeScript, React Router แยกเส้นทางต่อเกม
- **ฐานข้อมูล:** Cloud Firestore (Firebase) — หลายเครื่องเห็นข้อมูลชุดเดียวกัน อัปเดตเรียลไทม์ผ่าน snapshot listener, มี offline persistence ในตัว (เน็ตหลุดแล้วกลับมา ข้อมูลตามให้เอง)
- **Hosting:** Firebase Hosting (ฟรี)
- **i18n:** react-i18next, ไฟล์แปล `th.json` / `en.json`, เก็บภาษาที่เลือกใน localStorage
- **ไม่มีเซิร์ฟเวอร์ backend ของตัวเอง** — เว็บคุยกับ Firestore ตรง ๆ ควบคุมสิทธิ์ด้วย Firestore Security Rules

### การป้องกันด้วย PIN

- ตอนสร้างทัวร์นาเมนต์ ผู้จัดตั้ง PIN; เว็บเก็บเป็นแฮช (SHA-256) ในเอกสารทัวร์นาเมนต์
- เข้า "โหมดกรรมการ" โดยใส่ PIN ถูกต้อง เว็บจำสถานะไว้ใน sessionStorage (ปิดแท็บแล้วหาย)
- Firestore Security Rules ตรวจทุก write: เอกสารที่ส่งมาต้องมี field `_pinHash` และค่าต้องตรงกับ `pinHash` ของเอกสารทัวร์นาเมนต์แม่ (rules ใช้ `get()` อ่านมาเทียบ) — ไม่ตรงถูกปฏิเสธ
- ระดับความปลอดภัย: กันมือซนทั่วไป เพียงพอกับงานโรงเรียน/ชมรม ไม่ออกแบบสำหรับข้อมูลลับ

## 4. โครงสร้างข้อมูล (Firestore)

```
tournaments (collection)
  {tournamentId}
    name: string                         // "แข่งเอแมทประจำปี 2569"
    game: "thai-chess" | "chess" | "go" | "checkers" | "crossword" | "amath" | "boardgame"
    format: "free" | "round-robin" | "swiss" | "knockout"
    scoring: { win: number, draw: number, loss: number }   // ค่าเริ่มต้น 1 / 0.5 / 0
    pinHash: string
    status: "active" | "finished"
    createdAt: timestamp

    players (subcollection)
      {playerId}: { name: string, seed: number }           // 1–50 คน

    matches (subcollection)
      {matchId}:
        round: number                    // รอบที่ (บันทึกอิสระใช้ round = 0)
        player1Id: string
        player2Id: string | null        // null = บาย (bye)
        result: "p1win" | "draw" | "p2win" | "pending"
        rawScore1?: number              // คะแนนดิบ ไม่บังคับ
        rawScore2?: number
        createdAt: timestamp
```

หลักการสำคัญ: **ตารางคะแนน/อันดับไม่เก็บในฐานข้อมูล** — คำนวณสดฝั่งเว็บจาก matches ทุกครั้ง จึงไม่มีปัญหาข้อมูลค้าง/ไม่ตรงกัน และแก้ผลย้อนหลังแล้วอันดับถูกต้องเสมอ

## 5. หน้าเว็บและเส้นทาง (Routes)

| เส้นทาง | หน้า |
|---|---|
| `/` | หน้าแรก: การ์ด 7 หมวดเกม + จำนวนทัวร์นาเมนต์ที่กำลังแข่งของแต่ละหมวด |
| `/:game` | รายการทัวร์นาเมนต์ของเกมนั้น แยก "กำลังแข่ง" / "จบแล้ว" + ปุ่มสร้างใหม่ |
| `/:game/new` | ฟอร์มสร้างทัวร์นาเมนต์ |
| `/:game/:tournamentId` | หน้าทัวร์นาเมนต์ (แท็บ: ตารางคะแนน / ผลการแข่ง / สายแข่ง* / ผู้เข้าแข่ง) |

\* แท็บสายแข่งแสดงเฉพาะ format = knockout (เฟส 2)

### ฟอร์มสร้างทัวร์นาเมนต์

ชื่อรายการ, ระบบจับคู่, แต้มชนะ/เสมอ/แพ้ (แก้ค่าเริ่มต้นได้), PIN, รายชื่อผู้แข่ง — พิมพ์ทีละคนหรือวางทั้งชุด (ระบบแยกตามบรรทัด), ตรวจ 1–50 คน

### หน้าทัวร์นาเมนต์

- **แท็บตารางคะแนน (ค่าเริ่มต้น):** อันดับ, ชื่อ, แข่งแล้ว, ชนะ/เสมอ/แพ้, คะแนนสะสม (+ คะแนนดิบรวม ถ้ามีการกรอก) — เรียลไทม์, เหมาะฉายขึ้นจอใหญ่
- **แท็บผลการแข่ง:** แมตช์จัดกลุ่มตามรอบ; โหมดกรรมการเห็นปุ่มกรอก/แก้ผล + ช่องคะแนนดิบ (ไม่บังคับ); ระบบอิสระมีปุ่ม "เพิ่มแมตช์" เลือกผู้เล่น 2 คน; พบกันหมดมีปุ่ม "สร้างตารางแข่ง" ครั้งเดียว
- **แท็บผู้เข้าแข่ง:** รายชื่อ; โหมดกรรมการเพิ่ม/ลบ/แก้ชื่อได้ (ลบได้เฉพาะคนที่ยังไม่มีแมตช์)
- **โหมดกรรมการ:** ปุ่มมุมขวาบน ใส่ PIN; แสดงสถานะชัดเจนว่าอยู่ในโหมดแก้ไข

### การออกแบบ

Mobile-first (กรรมการกรอกผลจากมือถือข้างโต๊ะแข่ง) และตารางคะแนนดูดีบนจอใหญ่; ปุ่มสลับ ไทย/EN มุมบนทุกหน้า

## 6. ลอจิกจับคู่และอันดับ

ลอจิกทั้งหมดเป็น pure function ในโมดูล `src/lib/` แยกจาก UI และ Firestore เพื่อทดสอบง่าย:

- **`roundRobin(players)`** — สร้างคู่ครบทุกคู่แบ่งรอบด้วยสูตร circle method; จำนวนคนคี่ได้บายรอบละคน (บายไม่ได้แต้ม ถือว่าไม่ได้แข่ง)
- **`swissPair(players, history, standings)`** (เฟส 2) — จับคู่คะแนนใกล้กัน ไม่ซ้ำคู่เดิม คนคี่ได้บาย (ได้แต้มเท่าชนะ ตามธรรมเนียมสวิส); กรรมการกดสร้างทีละรอบ
- **`knockoutBracket(players)`** (เฟส 2) — สร้างสายแข่งขนาดกำลังสองที่ใกล้ที่สุด คนเกินได้บายรอบแรกตาม seed; ผู้ชนะเลื่อนสายอัตโนมัติเมื่อกรอกผล; เสมอในน็อกเอาต์ระบบเตือนให้แข่งใหม่หรือกรรมการชี้ขาด (ต้องเลือกผู้ผ่านเข้ารอบ)
- **`computeStandings(players, matches, scoring)`** — คืนอันดับเรียงตาม:
  1. คะแนนสะสม (ตาม scoring ของทัวร์นาเมนต์)
  2. ผลเจอกันตรง (head-to-head) เฉพาะกรณีเท่ากัน 2 คนและเคยเจอกัน
  3. คะแนนดิบรวม — รวมเฉพาะแมตช์ที่กรอกไว้ แมตช์ที่ไม่กรอกนับเป็น 0 ทั้งสองฝ่าย
  4. ยังเท่ากัน → อันดับร่วม

## 7. กรณีขอบและการป้องกันข้อผิดพลาด

- จำนวนผู้เล่นบังคับ 1–50; ฟอร์มเตือนเมื่อเกินหรือว่าง
- ชื่อผู้เล่นซ้ำในทัวร์นาเมนต์เดียวกัน → เตือนและให้ยืนยันก่อน (อนุญาตได้ เพราะบางกรณีชื่อจริงซ้ำกัน)
- แก้ผลย้อนหลังได้เสมอ ตารางคะแนนคำนวณใหม่ทันที; ในสวิส/น็อกเอาต์ ถ้ารอบถัดไปถูกสร้างแล้ว ระบบเตือนว่าการแก้ผลจะไม่ย้อนแก้การจับคู่ที่เกิดไปแล้ว
- ลบทัวร์นาเมนต์: ต้องอยู่ในโหมดกรรมการ + พิมพ์ชื่อทัวร์นาเมนต์ยืนยัน
- เพิ่มผู้เล่นหลังสร้างตารางพบกันหมดแล้ว → ระบบเสนอสร้างแมตช์เพิ่มเฉพาะคู่ที่ขาด
- เน็ตหลุดขณะกรอก: ใช้ Firestore offline persistence เขียนลงเครื่องก่อน ซิงก์เมื่อเน็ตกลับมา
- ปิดทัวร์นาเมนต์ (status = finished) แล้วผลทั้งหมดล็อก อ่านได้อย่างเดียว (เปิดกลับมาแก้ได้ด้วย PIN)

## 8. การทดสอบ

- **Unit test (Vitest):** ครอบคลุม `roundRobin`, `swissPair`, `knockoutBracket`, `computeStandings` — กรณีคนคี่/คู่, 1 คน, 50 คน, คะแนนเท่ากันหลายชั้น, บาย, แก้ผลย้อนหลัง
- **ทดสอบ UI:** ใช้งานจริงผ่าน browser preview ตาม flow หลัก: สร้างทัวร์นาเมนต์ → กรอกผล → ดูอันดับ → สลับภาษา → ทดสอบ PIN ผิด/ถูก
- **Firestore Rules:** ทดสอบด้วย Firebase emulator ว่า write โดยไม่มี PIN ถูกปฏิเสธ
